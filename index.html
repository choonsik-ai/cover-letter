1. 지원 동기 및 포부: 대규모 트래픽 속에서도 흔들리지 않는 톱니바퀴

"순간적인 트래픽 폭주에도 굳건한 [지원 회사명]의 아키텍처에 기여하다"

백엔드 엔지니어의 진짜 가치는 '평상시'가 아닌 '위기 상황'에서 증명된다고 생각합니다. [지원 회사명]은 최근 대규모 프로모션과 선착순 이벤트 등에서 발생하는 초당 수만 건의 트래픽(TPS)을 안정적으로 처리하며 사용자 경험을 방어해 냈습니다. 저는 이처럼 겉으로 드러나지 않는 곳에서 분산 환경과 데이터 정합성을 치열하게 고민하는 조직에서 제 백엔드 역량을 꽃피우고 싶습니다.

지금까지 Java와 Spring Boot를 기반으로 개발하며, 단순히 기능 구현에 만족하지 않고 "데이터가 100배 많아진다면?", "동시에 1,000명의 사용자가 접근한다면?"과 같은 극단적인 상황을 가정하며 코드를 작성해 왔습니다. [지원 회사명]이 직면한 대용량 데이터 처리와 아키텍처 고도화 과정에 합류하여, 비즈니스의 성장을 뒷받침하는 가장 단단한 톱니바퀴로 성장하겠습니다.

2. 직무 역량: 100만 건 데이터 통계 조회의 병목 해결과 Query 최적화

"추상적인 아키텍처를 넘어, 실제 데이터베이스의 성능을 튜닝한 경험"

사내 어드민 페이지 구축 프로젝트를 진행하며, 약 100만 건의 결제/주문 이력 데이터를 다루는 과정에서 심각한 성능 저하(API 응답 속도 약 3.5초)를 경험했습니다. 원인을 분석한 결과, JPA의 N+1 문제와 다중 필터(기간, 상태, 카테고리 등) 적용 시 발생하는 풀 테이블 스캔(Full Table Scan)이 문제였습니다.

이를 해결하기 위해 첫째, Fetch Join을 적용하여 불필요한 추가 쿼리 발생을 막아 N+1 문제를 해결했습니다. 둘째, 복잡한 동적 쿼리를 처리하기 위해 QueryDSL을 도입하고, 조회 조건에 맞춰 복합 인덱스(Composite Index)를 재설계했습니다. 또한, 페이징 처리 시 발생하는 Count 쿼리의 부하를 줄이기 위해 No-Offset 방식을 적용했습니다.

그 결과, 어드민 페이지의 통계 API 응답 속도를 기존 3.5초에서 0.2초로 약 94% 개선할 수 있었습니다. 이 경험을 통해 백엔드 로직의 구조화(Layered Architecture)뿐만 아니라, 실제 DB의 실행 계획(Explain)을 분석하고 쿼리를 최적화하는 'DB 중심적 사고'의 중요성을 깨달았습니다.

3. 트러블슈팅 경험: 선착순 이벤트의 동시성 이슈 해결 및 무결성 보장

"Race Condition 해결을 위한 Redis 분산 락(Distributed Lock) 도입"

E-commerce 토이 프로젝트 진행 중, '선착순 100명 쿠폰 발급' 이벤트를 구현하며 동시성 문제(Race Condition)를 겪었습니다. JMeter를 활용해 500명의 가상 유저가 동시에 쿠폰 발급 API를 호출하는 스트레스 테스트를 진행한 결과, 초과 발급(초과 재고 감소)되는 심각한 데이터 무결성 훼손을 확인했습니다.

문제를 해결하기 위해 Java의 synchronized 키워드부터 DB의 비관적 락(Pessimistic Lock), 그리고 Redis를 활용한 분산 락까지 다양한 대안을 두고 Trade-off를 고민했습니다. 단일 서버 환경이 아닌 다중 서버 확장을 고려했을 때 synchronized는 한계가 있었고, 비관적 락은 DB 트랜잭션 경합으로 인한 성능 저하 우려가 있었습니다.

결과적으로 성능과 확장성을 모두 잡기 위해 메모리 기반인 Redis와 Redisson 라이브러리의 Pub/Sub 구조를 활용한 분산 락을 도입했습니다. 이를 통해 DB에 가해지는 부하를 최소화하면서도 100개의 쿠폰이 정확하게 발급되도록 동시성 제어를 완벽히 구현해냈습니다.

4. 성격의 장단점 및 협업 태도: 단순한 API 연동을 넘어선 성능 관점의 커뮤니케이션

"프론트엔드의 요구사항과 백엔드의 성능 사이에서 최적점을 찾다"

저의 강점은 '목적 지향적 커뮤니케이션'입니다. 프로젝트 당시 프론트엔드 팀원으로부터 특정 페이지 렌더링을 위해 5개 이상의 흩어진 데이터를 한 번에 조회해 달라는 요구(Complex Nested JSON)를 받았습니다. 당시 백엔드 DB 구조는 정규화가 강하게 되어 있어, 이를 실시간으로 조인하여 내려줄 경우 서버 리소스 소모가 크고 응답 속도가 저하될 것이 뻔했습니다.

저는 프론트엔드 개발자에게 서버 병목의 위험성을 설명하고, 대신 타협안을 제시했습니다. 실시간성이 덜 중요한 데이터는 읽기 전용 모델(Read Model DTO)로 분리하고 Redis에 10분 단위로 캐싱(Caching)하여 제공하는 방식이었습니다. 결과적으로 프론트엔드는 원하는 형태의 데이터를 한 번의 API 호출로 빠르게 응답받을 수 있었고, 백엔드 서버 역시 과도한 조인 연산의 부담을 덜 수 있었습니다. [지원 회사명]에서도 이처럼 직군의 경계를 넘어, 타 부서와 기술적인 맥락을 공유하며 프로덕트 관점의 최적 해를 찾아내는 개발자가 되겠습니다.
